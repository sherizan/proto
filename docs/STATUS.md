## Known issues / quirks

- **`pnpm create proto@latest` has 120-min stale-cache** for `@latest` resolution. Workaround: `pnpm create proto@<exact-version>` to force fresh fetch. `npm create proto@latest` doesn't have this issue.
- **3 pre-existing typecheck errors in `packages/proto-components`** (`Card.tsx` can't find `expo-glass-effect` types; `Toggle.tsx` can't find `@expo/ui/swift-ui` types + has one implicit-any). Not caused by the motion-primitives work — present on `main` before this PR. Not blocking; the modules still compile and run.
- **iOS Simulator can stash old Prototo binaries across device instances.** If a designer's `proto start` boots a different simulator device than where the new Prototo was installed, they see a stale version. `ensure-prototo-app` SHA-checks via the GitHub Release manifest and auto-reinstalls when out of date, but only on `proto start` — the first launch of an unused simulator may need `rm -rf ~/.prototo/cache` + a fresh `proto start`.
- **Verifying a native module is in the dev client: check `Prototo.debug.dylib`, NOT `Frameworks/`.** Most Expo/RN modules are statically linked into `Prototo.app/Prototo.debug.dylib` (the ~53MB debug dylib), not shipped as separate `.framework`s — only some (ExpoImage, ExpoFileSystem, ExpoModulesCore, ExpoCamera, ExpoMaps, ExpoVideo, RNReanimated, RNScreens, RNSkia, RNSVG, async-storage, safe-area) get their own framework. The main `Prototo` executable is a ~120KB launcher shell with no module code. To confirm a module: `strings -a Prototo.app/Prototo.debug.dylib | grep -c '<ModuleName>'` (e.g. `ExpoAsset` / `AssetModule`). This was the trap during the 0.1.11 hotfix — the framework list falsely implied the fix failed.
- **`npm cache clean --force` prints an ownership warning** (`sudo chown -R 501:20 /Users/sherizan/.npm`). Pre-existing on Sheri's machine from a past `sudo npm`. Cosmetic.
- **A leftover local Xcode App-Clip build can shadow the official dev client (resolved 2026-06-14).** During the Phase E / E2 on-device tests, clip-containing builds were installed via local Xcode under the **same bundle id `com.sherizan.prototo`** (Phase E embedded the clip into `apps/prototo-app`) and `com.sherizan.prototo.viewer` (Phase E2). Those locally-installed binaries override the official one by bundle id, so the device shows the "App-Clip version" even though the official channels never shipped a clip build. **The official builds are clean:** install URL `prototo.app/install/ios/app` → device build `3b56455a` (commit `b5918f9f`, plain SDK-56 dev client, no clip); the clip-era device build `3d4e1770` ERRORED and never shipped; the only de-clipped finished build is the sim `cf91c268` (on Appetize). **Fix:** delete Prototo (and any "Prototo Viewer") from the device, reinstall from `prototo.app/install/ios`. Confirmed working.
